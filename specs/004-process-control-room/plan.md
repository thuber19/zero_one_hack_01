# Implementation Plan: Process Control Room GUI

**Branch**: `khaled_experiments` | **Date**: 2026-05-30 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `/specs/004-process-control-room/spec.md`

---

## Summary

Build a dark-mode, data-dense React SPA that gives Infineon process engineers a real-time window into the fab-yield model trained in Spec 001/002. The frontend (Vite 5 + React 18 + TypeScript 5 + Tailwind 3 + shadcn/ui) is hosted on Vercel and calls a FastAPI backend (Fly.io/Render) that wraps frozen Spec 001 model inference and server-side SHAP computation. Five views — Wafer Journey (D3), KPI Strip, SHAP Panel, Sequence Optimizer (dnd-kit), Batch Inspector — are built in priority order (P1 first) and driven by hard-coded demo fixtures so the demo is de-risked against backend unavailability.

**Primary Dependencies**:
- Frontend: Vite 5, React 18, React Router 6, TanStack Query 5, Zustand 4, Tailwind CSS 3, shadcn/ui, Recharts 2, D3 v7, dnd-kit 6
- Backend: FastAPI 0.110+, Pydantic v2, Uvicorn, PyTorch (Spec 001 checkpoint), SHAP library, numpy, pandas, `cachetools` (LRU)

**Testing**: Vitest + React Testing Library (frontend unit/component tests); pytest (backend unit tests for SHAP services and route contracts)

**Target Platform**: Desktop browser (Chrome/Firefox), 1920×1080 or 2560×1440. No mobile support required.

**Project Type**: Web application — SPA frontend + REST API backend

**Performance Goals**:
- `/predict` and `/optimize` p95 < 2,000 ms end-to-end (browser `fetch()` start → JSON parse complete)
- Wafer Journey initial paint of 100-step batch < 500 ms from API response
- KPI strip count-up animation at ≥ 60 fps (4 simultaneous numbers)
- D3 SVG frame budget ≤ 16 ms per step node during paint for up to 300 steps

**Constraints**:
- All API calls have a 5-second client-side timeout; UI retains last valid state and shows non-blocking toast on timeout
- Loveable vibe-coding platform compatible (Vite + React + TypeScript + Tailwind; no SSR, no Next.js)
- No authentication layer
- Supabase is optional; `VITE_SUPABASE_URL` guards all Supabase calls
- `/predict` response cached in-memory LRU on the server (key: `batch_id`); SHAP background precomputed at startup

**Scale/Scope**: 5 views, ~15 components, 5 API endpoints, 2 fixture files (bad batch + good batch), target 3-minute demo flow

## API Endpoints — Concrete Signatures

### `POST /predict`

**Decision: Synchronous REST** — sub-2s budget met by pre-loaded model + SHAP background + LRU cache. No streaming / SSE. If inference exceeds 2s post-demo, refactor to `POST /predict/async` → poll `GET /predict/{job_id}` (out of scope for v1).

## SHAP Strategy

**Explainer choice: `DeepExplainer`** (model-specific, ~10× faster than `KernelExplainer` for neural nets).
- Fallback to `KernelExplainer` if `DeepExplainer` raises (e.g., model architecture not supported).
- SHAP background dataset: 100 representative training sequences, precomputed offline on Leonardo as `api/app/shap_background.npy` and loaded once in the `lifespan` handler.
- Per-batch SHAP values cached in `cachetools.LRUCache(maxsize=256)` keyed by `batch_id` (or MD5 hash of serialized sequence for anonymous requests).
- JSON shape shipped to frontend: `list[ShapContribution]` per step — frontend renders waterfall and beeswarm from this flat array.
- SHAP computation is server-side only; frontend receives pre-computed values.

**Startup pre-warm sequence** (in `lifespan` handler):
1. Load Spec 001 checkpoint into model (`torch.no_grad()` mode).
2. Load `shap_background.npy`.
3. Initialize `DeepExplainer(model, background)` — stored as module-level singleton.
4. Run a single dummy inference pass through `/predict` with the bad-batch fixture to populate initial LRU cache and verify end-to-end path.
5. Log startup time and `GET /health` returns `status=ok`.

## Component Breakdown

### `WaferJourney.tsx` — D3 v7, custom SVG

- Uses a `useRef<SVGSVGElement>` and a dedicated `useEffect` for D3 rendering (avoids React reconciliation conflicts — D3 owns the SVG DOM subtree entirely).
- Pattern: React renders `<svg ref={svgRef} />` shell; D3 effect runs `d3.select(svgRef.current)` and manages all child nodes.
- On `per_step` data change: D3 `join()` pattern for enter/update/exit (no full re-render).
- Step nodes: `<circle>` colored by `risk_score` threshold (red ≥ 0.85, amber 0.70–0.84, neutral otherwise). Accent hex values sourced from CSS variables matching Tailwind config.
- Horizontal scroll: `<div style={{ overflowX: 'auto' }}>` wrapping the SVG; SVG width = `stepCount * NODE_SPACING` (auto-sized).
- Performance for 300 steps: use D3 `scaleLinear` for x-positions; batch DOM mutations; no layout thrashing. Virtualize only if profiling shows jank beyond 300 steps.
- Popover: `StepPopover.tsx` — a `<div>` positioned via D3 mouse coordinates, rendered into a React portal; shown on `mouseover`, hidden on `mouseleave`. Popover content fetches from Zustand `selectedStep` state.

### `KpiStrip.tsx` — persistent top bar

- Mounted once in `App.tsx` layout, never unmounted during navigation (React Router `<Outlet>` is below it).
- Four `<KpiCard>` sub-components, each receiving `label`, `value`, `unit`, `colorClass`.
- Delegates animation to `useCountUp(targetValue, durationMs=800)` hook.
- `useCountUp`: uses `requestAnimationFrame` loop with easing function; cancels on unmount; if tab is backgrounded, completes instantly on re-focus via `document.visibilitychange` listener.
- Color logic: yield ≥ 0.85 → `text-green-400`; yield < 0.70 → `text-red-400`; else `text-amber-400`. All other KPIs use neutral until threshold crossed.
- All numeric values displayed in monospace font (`font-mono` Tailwind class).

### `ShapPanel.tsx` — waterfall + beeswarm

- **Waterfall chart**: Recharts `BarChart` with custom `Cell` fill (green if `contribution > 0`, red if `< 0`). Bars sorted by `Math.abs(contribution)` descending; top 10 shown. Feature labels sourced from `featureNameMap.ts`; unknown features fall back to title-cased, underscore-stripped name.
- **Beeswarm**: Custom SVG drawn with `d3.forceSimulation` (x = contribution magnitude, y = jittered by force). Rendered inside a `<svg>` ref managed by D3 (same pattern as WaferJourney). Small (200×100 px) — secondary visual.
- **Transitions**: On step change, Recharts re-renders with new data (300 ms CSS transition on bar width via `animationDuration={300}`). Beeswarm D3 re-simulates with 300 ms transition.
- **No SHAP data**: `shap` is null or empty array → render placeholder `<p>SHAP data not available for this step</p>`. No chart, no error.
- Driven by Zustand `selectedStep` slice; panel updates whenever `selectedStep` changes.

### `SequenceOptimizer.tsx` — dnd-kit drag-reorder

- Uses `@dnd-kit/core` `DndContext` + `@dnd-kit/sortable` `SortableContext` + `useSortable` per step.
- Drag-end handler: updates Zustand `dragState`; schedules debounced (200 ms) call to `useOptimize` hook.
- `useOptimize`: TanStack Query `useMutation` that POSTs `{ sequence: step[] }` to `/optimize`. On success, updates displayed yield. On timeout (> 5s) or network error, shows `sonner` toast ("Optimizer unavailable — try again") and retains previous yield.
- Optimistic UI: yield score shows spinner immediately on drag-end; replaced by real value on response.
- Non-standard order badge: if `OptimizeResponse` includes step at an illegal position (detected by comparing sequence against a bundled `canonical_order` fixture), render a small `<Badge variant="outline">Non-standard order</Badge>` (shadcn Badge). Drag is never blocked.
- Sequence length cap: drag UI supports ≤ 100 steps (spec assumption). Longer sequences are read-only in `BatchInspector`.

### `BatchInspector.tsx` — shadcn Table + Drawer

- `useBatches` hook: TanStack Query with query key `['batches', { page, sortBy, order }]`. Cache per page independently — navigating to page 3 does not re-fetch pages 1–2.
- shadcn `<Table>` with sortable column headers. Click header → toggle sort; second click → reverse order → updates query key → new fetch.
- Row with `defect_probability >= 0.80`: rendered with `border-l-4 border-red-500` (red left border).
- Page size: 20 rows. Pagination controls: shadcn `<Button>` prev/next + page indicator.
- Empty state: `<div>No batches found</div>` with appropriate icon when `total === 0`.
- Click row → opens shadcn `<Drawer>` (side, right). Drawer triggers `useBatchDetail(batchId)` query (`['batch', batchId]`). Inside drawer: batch metadata + mini per-step risk strip (simplified `WaferJourney` variant without hover — just colored dots for each step).

|-----|--------|------------|

### Error / Fallback Flow

1. `GET /health` returns `status=degraded` or times out → `setApiAvailable(false)` → `<ErrorBanner>` mounts ("Model API unavailable — displaying cached data").
2. `POST /predict` fails → TanStack Query `onError` → toast; stale fixture data shown via `initialData` seeded from `lib/fixtures.ts`.
3. `POST /optimize` times out (> 5s) → `sonner` toast; previous yield retained.
4. Browser tab re-focus after backgrounded count-up: `useCountUp` checks `document.hidden` on `visibilitychange`; if animation was in-flight, jumps to final value immediately.

## Performance Constraints

### Backend

- **Model pre-warm**: Spec 001 checkpoint loaded once in `lifespan`; kept as module-level singleton. No weight reload per request.
- **SHAP background**: Loaded from `.npy` once; `DeepExplainer` initialized once; both held in app state via FastAPI `Request.app.state`.
- **LRU cache**: `cachetools.LRUCache(maxsize=256)` for `/predict` responses (key: `batch_id`) and `/optimize` responses (key: MD5 of serialized sequence). Prevents re-running inference for repeated demo requests.
- **Uvicorn workers**: Single worker (`--workers 1`) sufficient for demo; avoids cache invalidation across processes. Increase for production if needed.

### Frontend

- **D3 WaferJourney**: D3 owns SVG DOM subtree; React renders only the shell. Avoids React reconciliation overhead on 300 nodes. If profiling shows jank, chunk into rows of 50 nodes rendered in separate `<g>` elements.
- **KPI count-up**: Pure `requestAnimationFrame` loop; no `setInterval`, no React state updates during animation (writes directly to DOM via `ref`). This avoids React re-renders during the 800 ms animation.
- **TanStack Query `staleTime: Infinity`** for `predict` and `batch` detail queries — no background re-fetching during demo.
- **Bundle size**: D3 imported tree-shakeable (`import { select, scaleLinear } from 'd3'`). Recharts tree-shaken via named imports. Target total bundle < 500 KB gzipped.

## Complexity Tracking

| Concern | Why Needed | Simpler Alternative Rejected Because |
|---------|------------|--------------------------------------|
| D3 v7 for WaferJourney (alongside Recharts) | Recharts cannot render 300 arbitrary SVG nodes with D3 force-layout for beeswarm or custom step-node geometry | Recharts ScatterChart lacks the freeform SVG control and D3 force simulation needed for the Wafer Journey aesthetic |
| dnd-kit for SequenceOptimizer | Native HTML5 drag-and-drop has poor accessibility and inconsistent cross-browser behavior; `react-beautiful-dnd` is unmaintained | No maintained, Loveable-compatible DnD library with sortable semantics other than dnd-kit |
| Zustand (alongside TanStack Query) | Server state (TanStack Query) and ephemeral client state (selected step, drag in-progress) are genuinely different concerns | TanStack Query `useQueryClient` would require global cache mutation for UI-only state like `selectedStepId`, which pollutes the query cache |

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| D3 + React reconciliation conflict | Medium | High (janky WaferJourney) | D3 owns SVG subtree entirely; React renders shell only. Pattern is established before Phase 1 ends. |
| SHAP latency > 2s | Medium | High (breaks SC-001) | DeepExplainer (10× faster than KernelExplainer); background pre-loaded; LRU cache covers repeat demo requests. Profile in Phase 2; fallback to approximate SHAP if needed. |
| Model cold start on Fly.io/Render | Medium | Medium (slow first request) | Pre-warm on `lifespan` startup; Fly.io `min_machines_running=1` to prevent scale-to-zero. |
| Loveable export friction | Low | Medium (demo iteration speed) | Scaffold and design tokens committed to repo; Loveable imports from repo. Test Loveable import in Phase 0. |
| Spec 001 checkpoint not available at GUI demo time | Low | High (no real inference) | Fixture-first design: full demo works with `demoMode=true` and no backend. |
| Wafer Journey jank at 300 steps | Low | Medium | D3 join() pattern; no React re-render on hover. Profile in Phase 1; virtualize if needed. |
